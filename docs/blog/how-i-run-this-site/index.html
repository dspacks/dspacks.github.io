How I Run This Site with Ghost, GitHub Pages, and LLMs

The stack is simple on purpose: Ghost for drafting, GitHub Pages for delivery, and git-controlled HTML for everything else.

The short version

I wanted a site that could hold a normal portfolio, a blog, and a growing pile of small HTML apps without turning into a maintenance project. The answer was to keep the public site static, keep the content in git, and let a few scripts do the repetitive work.

I also wanted a workflow that works well with LLMs. I use them to draft pages, generate small tools, tighten copy, and update indexes — but the repo still stays in charge. If it is not committed, it does not ship.

What lives where

Piece Role
GhostDrafting and long-form publishing for blog posts or CMS-managed pages
docs/Live GitHub Pages output, including the home page, projects, apps, and blog
scripts/export.shCrawls local Ghost and exports it into static files under docs/
scripts/index-apps.shScans docs/apps/ and regenerates the app gallery + manifest
scripts/index-blog.shRegenerates the blog landing page + manifest from post folders
LLMsDrafting, editing, app generation, and content cleanup

The publishing flow

  1. Write or revise the content locally.
  2. For a blog post, work in Ghost or create a folder under docs/blog/.
  3. For a mini app, create a folder under docs/apps/<name>/ with an index.html.
  4. Run the indexers so the gallery and blog landing page stay in sync.
  5. Commit, push, and let GitHub Pages deploy the result.

The key point is that I do not hand-edit a central index every time I add something. The repo discovers it for me.

How the app gallery works

The app gallery is the part that makes this useful for random HTML tools. I can ask an LLM to build a single-file app, drop it in docs/apps/some-tool/index.html, and the gallery picks it up automatically.

Each app can declare metadata in the head: 

<meta name="app:name" content="My Tool" />
<meta name="app:description" content="Short summary" />
<meta name="app:tags" content="gemini,html,utility" />
<meta name="app:date" content="2026-06-24" />

The generator reads those tags, builds the gallery card, and writes a manifest. That means the site can grow by addition instead of by constant curation.

How I keep drafts out of the way

I also have a hidden-content convention now, because starter pages and placeholders are easy to accidentally publish. Anything marked as a template, draft, or placeholder gets excluded from the generated indexes. 

<body class="is-template" data-status="template">
<meta name="robots" content="noindex, nofollow" />

That lets me keep starter content in the repo without making it visible on the live site.

LLMs as co-managers, not autopilots

I use LLMs in three ways:

  • Drafting: turning rough notes into a publishable page
  • Transformation: converting an idea into a working HTML app or product page
  • Maintenance: updating galleries, metadata, contrast, copy, and layout details

But the important constraint is that git is still the source of truth. The model can suggest structure, but the repo decides what is live.

Why this setup works for me

I do not want a heavyweight CMS for everything, and I do not want a pile of one-off documents that drift out of sync. This setup gives me a clean division:

  • Ghost for writing
  • GitHub Pages for publishing
  • Scripts for indexing and export
  • LLMs for accelerating the boring parts

It is simple enough to keep moving, but structured enough that I can add weird little tools without breaking the site.

Recent proof that it works

The best example is Sectograph. It started as a watchface project, moved through a product-style page, and is now published in the Pebble app store. That path was only practical because the site stack makes publishing cheap.

The site can support a normal portfolio, a shipping app page, a blog post, and a random HTML experiment — all from the same repo.

What I still want to improve

The next step is probably better cross-linking: a dedicated index for drafts, a stronger way to surface app release notes, and a planned migration path for older WordPress content from my backup when I’m ready to bring it over.

But the core idea is already working: keep the public site static, keep the content in git, and let LLMs help without taking control.
